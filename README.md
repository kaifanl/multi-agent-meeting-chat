**项目名称：** AI会议纪要助手 (基于多智能体架构与RAG问答)
**团队人数：** 4人 (P1, P2, P3, P4)
**项目目标：**
1.  设计并开发一个能够接收会议音频（或文本），通过多智能体协作，自动生成结构化会议纪要的AI系统。纪要应包含关键议题、决策点、行动项等。
2.  实现一个基于RAG (Retrieval Augmented Generation) 的问答功能，允许用户针对生成的会议纪要内容进行提问。

**核心智能体角色设想（与原计划基本一致，略作调整）：**
1.  **协调与接口智能体 (Coordinator & Interface Agent):** (P1)
    *   接收用户输入（音频/文本，用户提问）。
    *   调用其他智能体，管理任务流程和数据流。
    *   向用户呈现最终的会议纪要和问答结果。
2.  **语音转录智能体 (Transcription Agent):** (P2)
    *   负责将会议音频转换为文本。
3.  **内容分析与提取智能体 (Analysis & Extraction Agent):** (P3)
    *   接收转录后的文本。
    *   识别核心议题/主题。
    *   提取关键决策点。
    *   识别并提取行动项。
    *   对议题进行简要总结。
4.  **纪要生成与格式化智能体 (Minutes Generation & Formatting Agent):** (P4)
    *   整合来自内容分析智能体的结构化信息。
    *   按照预设模板生成格式化的会议纪要。
5.  **RAG问答智能体 (RAG Q&A Agent):** (P2 或 P3 负责，P1辅助集成)
    *   接收生成的会议纪要作为知识源。
    *   接收用户问题，从知识源中检索相关信息。
    *   结合检索到的信息和用户问题，通过LLM生成回答。

**技术栈建议（RAG补充）：**
*   **向量嵌入 (Embeddings):** OpenAI `text-embedding-ada-002` API (或其他轻量级本地模型如Sentence Transformers，但API更快速集成)。
*   **向量存储/检索 (Vector Store/Retrieval):**
    *   **极简方案:** 直接使用Python列表存储文本块，用余弦相似度计算与问题向量的相似度进行检索（适合小文本量）。
    *   **轻量级库:** FAISS (CPU版本安装简单), ChromaDB (内存模式或简单文件存储)。避免需要复杂配置的数据库。
    *   **如果时间非常紧张:** 可以将整个会议纪要文本作为上下文直接喂给LLM进行问答，但效果可能不如RAG，且受限于上下文长度。

**负责人分工调整建议：**
*   **P1 (Lead/Coordinator & Integration):** 负责Coordinator Agent，整体系统集成，最终的UI/交互（如果做），以及报告和视频的整合与主导。
*   **P2 (Transcription & RAG Core):** 负责Transcription Agent，以及RAG问答智能体的核心技术实现（Embedding, 检索逻辑, LLM调用生成答案）。
*   **P3 (Analysis & Extraction & RAG Support):** 负责Analysis & Extraction Agent，设计核心的提取Prompt。同时，在RAG部分，协助P2理解纪要内容结构，为文本切分和检索提供支持。
*   **P4 (Formatting & Documentation/Testing):** 负责Formatting Agent，以及技术报告中部分章节的撰写、演示视频的辅助准备、测试用例的编写与执行。

**重要注意事项与风险（针对RAG）：**
1.  **RAG复杂度：** 实现一个基础的RAG系统会增加工作量，尤其是在文本切分、嵌入选择、检索策略和生成Prompt方面。需要快速决策并选择最简单的可行方案。
2.  **文本切分策略：** 如何将会议纪要切分成有意义的块 (chunks) 以便检索，这是一个关键点。可以按段落、固定长度或基于语义进行切分。
3.  **检索效果：** 简单的检索可能无法完美定位到最相关的信息，影响问答质量。
4.  **上下文长度：** 即使使用了RAG，喂给LLM生成答案的上下文长度也需要注意，避免超出模型限制。
